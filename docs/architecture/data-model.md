## Better Auth-owned tables

Generated by `@better-auth/cli generate` from [packages/auth/auth-cli.config.ts](../../packages/auth/auth-cli.config.ts) and committed into `schema.prisma`. Re-run the CLI when upgrading better-auth and diff the output before committing.

| Table          | Purpose                                                                          |
| -------------- | -------------------------------------------------------------------------------- |
| `user`         | Identity record. Email is unique (case-insensitive via app logic).               |
| `session`      | Long-lived session. `activeOrganizationId` is the bridge to RLS.                 |
| `account`      | OAuth/social/credential providers attached to a user.                            |
| `verification` | Token store for email verification, magic links, password reset.                 |
| `organization` | Tenant record. Foundation extends with `legalName`, `countryCode`, `austriaUid`. |
| `member`       | Many-to-many between users and organisations, with a role.                       |
| `invitation`   | Pending invitations to join an organisation.                                     |
| `twoFactor`    | TOTP secrets and backup codes (UI deferred).                                     |

These tables do **not** have RLS in Phase 2. Tenant access on `organization`/`member`/`invitation` is enforced by Better Auth's organization plugin permission system. See [rls-strategy.md](rls-strategy.md) for the rationale and the plan to add defence-in-depth RLS later.

## Privileges

| Role               | Tables (Better Auth + Foundation) | `audit_entry`       |
| ------------------ | --------------------------------- | ------------------- |
| `dutyhive_app`     | SELECT, INSERT, UPDATE, DELETE    | SELECT, INSERT only |
| `dutyhive_migrate` | owner — full DDL + DML            | full                |

The `dutyhive_app` REVOKE on `audit_entry` for UPDATE/DELETE is the database-level R-0004 mitigation: even if the application is compromised, the auth role cannot tamper with audit history. See migration `20260504184311_audit_entry_revoke_writes`.

## Re-generating the Prisma client

```bash
pnpm --filter @dutyhive/db exec prisma generate
```

Output lands in `node_modules/.prisma/client` at the workspace root (see schema.prisma's `output` directive — the path threads through pnpm's hoisting layout). The runtime `import { prisma } from '@dutyhive/db'` resolves through this generated client.

## Re-generating Better Auth tables

```bash
pnpm --filter @dutyhive/auth exec better-auth generate \
  --config auth-cli.config.ts \
  --output ../db/prisma/schema.prisma
```

Then diff `schema.prisma`, re-add Foundation-eigene fields and relations the CLI may have wiped, run `pnpm --filter @dutyhive/db exec prisma format`, and create a migration with `prisma migrate dev --name <descriptive_name>`.
